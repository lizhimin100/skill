## 三、🎯 主智能体（编排者）定义

### 角色定位
- **对应文件**：`orchestrator.md`（新建）
- **核心职责**：分配任务 → 接收结果（仅 PASS/FAIL + 文件路径）→ 写 main-log.md
- **思维模型**：建筑项目经理——不自己砌砖，只管分配工种、检查进度、写日志
- **输出物**：`main-log.md`（项目级日志，记录每次任务分发的开始/结束/结果）

### 工作流（参照视频 sub-agent 编排模式）

```
🎯 主智能体 启动
  │
  ├─ 读取 NOVEL_ORCHESTRATION.md（总文档）→ 确认当前阶段
  ├─ 读取 main-log.md → 确认上一个完成的任务
  │
  ├─ 【阶段判断】
  │   ├─ 项目未初始化 → 分发 🟢 项目启动官
  │   ├─ 无设定 → 分发 🟣 创意架构师
  │   ├─ 无大纲 → 分发 🔵 结构规划师
  │   ├─ 有章纲无正文 → 分发 🟠 叙事写手
  │   ├─ 有正文未审计 → 分发 🔴 红队检察官
  │   ├─ 有审计通过 → 分发 🟡 档案员
  │   └─ 档案员完成 → 分发 ⚪ 用户代理 预检下一章
  │
  ├─ 【分发任务】
  │   ├─ 只传递：输入文件路径 + 子智能体 skill 文件路径
  │   ├─ 不传递：任何文件的具体内容
  │   └─ 记录：{时间戳} 分发 [{子智能体名称}] → {输入路径}
  │
  ├─ 【等待完成 → 检查结果】
  │   ├─ 不读取子智能体的产出内容
  │   ├─ 只检查：文件是否存在？
  │   ├─ 如果是 🔴 红队：用 grep 提取第一行"### 判定" → PASS/FAIL
  │   ├─ 如果是 ⚪ 用户代理：提取分级颜色 🟢/🟡/🔴
  │   └─ 如果是 🟡 档案员：检查上下文档案是否存在
  │
  ├─ 【写日志】
  │   ├─ {yymmdd hhmm} [子智能体名称] 完成: {产出路径}
  │   ├─ {yymmdd hhmm} 判定: {PASS/FAIL}
  │   └─ {yymmdd hhmm} 当前进度: {第X章/第Y步}
  │
  └─ 【循环或等待】
      ├─ 全部完成 → 报告用户
      ├─ 红队 FAIL → 重新分发 🟠 写手（带审计报告路径）
      ├─ 用户代理 🔴阻断 → 暂停，等待用户确认
      └─ 有回退请求 → 执行回退流程
```

### 主智能体绝对禁止清单

1. **🔴 严禁** 主智能体读取子智能体的产出文件内容 —— 只检查文件存在性和 PASS/FAIL
2. **🔴 严禁** 主智能体直接编辑任何创作文件（正文/章纲/设定/上下文档案）
3. **🔴 严禁** 主智能体在当前会话中引用子智能体的推理过程
4. **🔴 严禁** 主智能体跳过日志记录 —— 每一步必须写入 main-log.md
5. **🔴 严禁** 主智能体在没有用户确认的情况下重新分发同一任务（防止死循环）

| 角色 | 颜色 | 实例化触发条件 | 实例化产物 |

5. **异常情况例外**：🟡 档案员更新"静态设定"时，必须在变更日志中注明触发者和原因

## 七、主智能体协作下的写作流程（V2 更新版）

### 🎯 主智能体写作循环（每章一次完整迭代）

```
🎯 主智能体：检查当前状态
  │
  ├─ 有章纲未写 → 分发 🟠 叙事写手
  │     🟠 前置读取清单（由主智能体指定路径）：
  │       1. SOLOENT.md §5（上下文档案索引）
  │       2. 上下文档案当前.md（续写指令 + 动态面板）
  │       3. 本章章纲.md
  │       4. SOLOENT.md §8（上一章结尾状态）
  │       5. 按需：静态设定文件
  │     🟠 完成正文 → 写入 5-正文/卷XX/第X章.md
  │     报告主智能体：完成 + 文件路径
  │
  ├─ 🟠 完成后 → 分发 🔴 红队检察官
  │     🔴 检查项：字数/爽点≥3/剧情点≥2（五段完整）/三线并行
  │     🔴 审计结论写入审计报告文件
  │     🔴 报告主智能体：✅ PASS 或 ❌ FAIL（只报告第一行判定）
  │     ├─ ✅ PASS → 继续
  │     └─ ❌ FAIL → 重新分发 🟠 + 提供审计报告路径 → 修复后再审
  │
  ├─ 🔴 通过后 → 分发 🟡 档案员
  │     🟡 更新上下文档案当前.md + 备份旧版本 + 更新变更日志
  │     🟡 报告主智能体：完成 + 上下文档案路径
  │
  ├─ 🟡 完成后 → 分发 ⚪ 用户代理
  │     ⚪ 预检下一章准备
  │     ⚪ 报告主智能体：🟢建议 / 🟡警示 / 🔴阻断
  │     ├─ 🟢建议 → 主智能体直接进入下一轮
  │     ├─ 🟡警示 → 主智能体暂停，等待用户书面确认
  │     └─ 🔴阻断 → 主智能体暂停，等待用户介入
  │
  └─ 🎯 写 main-log.md → 报告用户 → 等待用户决定
       ├─ 继续下一章
       ├─ 停下
       └─ 回退修复
```

## 九、行为约定：叙事写手的工作规则（无改动）

### 字数规则
- 每章 1800-3000 字为合理范围
- 超过 3000 字建议拆分两章
- 如果字数不足，**不强制 AI 注水**。剧情量不够时：
  - 标记"剧情量不足以支撑当前章节数"
  - 建议：补充剧情，或者减少一次生成的章节数

### 章纲结构规则（叙事写手的章纲拆解标准）
每章章纲必须包含：
1. **核心剧情**（大白话 + 三线并行标注）
   - 主线：主角的核心目标推进
   - 副线：其他角色的反应/行动/态度
   - 人物线：主角人设突出 + 关系发展
2. **至少 3 个爽点**（明确标注类型：对比爽/抽奖爽/能力爽/收获爽/打脸爽）
3. **至少 2 个完整剧情点**（第一章可例外）
   - 每个剧情点必须覆盖五段：
     - ① 危机/目标 → ② 考量/权衡（灵魂）→ ③ 行动/决策 → ④ 结果/转折 → ⑤ 收益/情报
4. **字数预期锚点**：1800-3000 字（不硬性限制，不足时退回补剧情）

## 十五、信标切换协议（全角色通用，V2 更新版）

**定义**：当工作从一个角色切换到另一个角色时，必须执行以下协议。在 V2 架构中，信标切换由 🎯 主智能体触发。

```
【信标·角色切换声明】
上一个任务（[任务名称]）的输出已定稿，文件已写入 [文件路径]。
我现在不再以[上一个角色名称]的身份思考。

开始工作。
```

**强制规则**：

5. **V2 增强**：🎯 主智能体在分发任务时，必须明确告知子智能体：
   - "你是一个新的推理实例"

   - "你只需读取指定输入文件，不要追问上一个子智能体的思考过程"

## 十六、⚪ 用户代理 分级响应机制（V2 更新版）

## 十七、回退逆流机制（V2 实现版 — 由 🎯 主智能体执行）

**触发条件**：
1. 🟠 叙事写手发现章纲无法支撑情绪目标
2. 🔴 红队检察官发现设定冲突
3. ⚪ 用户代理发出 🔴 阻断
4. 用户主动要求回退

**执行流程（V2 版）**：

```
[触发回退]
  ↓
触发者写入 _rollback_request.md（唯一路径：.novelkit/_rollback_request.md）
  ├─ 回退原因：[简述]
  ├─ 回退目标：[设定 / 大纲 / 正文 / 上下文档案]
  └─ 影响范围：[受影响章节/文件列表]
  ↓
🎯 主智能体 读取 _rollback_request.md
  ├─ 注意：这是主智能体唯一例外地读取子智能体产出内容的场景
  ├─ 记录到 main-log.md
  └─ 判断回退目标：
      ├─ 回退到 3-设定/3.2.1_静态设定/ → 分发 🟢 项目启动官 修正设定
      │   → 🟡 档案员 更新变更日志
      │   → 受影响章节标记
      ├─ 回退到 4-大纲/ → 分发 🔵 结构规划师 修正章纲
      │   → 标记受影响章节
      └─ 回退到 5-正文/ → 分发 🟠 叙事写手 重写
  ↓
🎯 主智能体 标记受影响章节到 main-log.md
  ↓
回退完成后 → 🎯 主智能体 恢复分发 → 报告用户
```

## 十九、设计原则与禁止清单（V2 更新版）

### 设计原则
1. **保留已有流程的核心逻辑**：改造不是重写，原有工作流骨架应保留
2. **禁止清单宁强勿弱**：将以往踩过的坑直接写成"严禁"，不要写"尽量避免"
3. **自检系统必须可执行**：写具体的检查项，而非"仔细检查"
4. **一次性只改造一个 skill**：按上述清单顺序，改完一个再改下一个
5. **SKILL.md 不宜过长**：超过 2000 tokens 时拆分到 reference/

### 全局禁止清单（V2 扩充——新增条目用 ★ 标注）

1. **严禁** 将"AI 自检"作为唯一审计手段 —— 必须由 🔴 红队检察官 外部审计
2. **严禁** 在设定变更时不记录变更日志 —— 每次变更必须写"变更原因 + 影响范围 + 日期"
3. **严禁** AI 擅自修改已确认的章纲 —— 章纲在 🔵 结构规划师 输出、🔴 审计、用户确认后即冻结
4. **严禁** 叙事写手在正文中引入新设定 —— 所有新设定必须先经过 🟣 创意架构师 流程
5. **严禁** 跳过红队审计 —— 任何节点的输出在交给用户确认前必须先经过 🔴 红队
6. **严禁** 使用"提示用户"作为审计不通过的最终处理方式 —— 不通过时 AI 必须先自行修复，再呈递给用户
7. **严禁** 将上下文档案中的数值与正文描述不一致时不标记 —— 🟡 档案员 必须捕获并标记
8. **严禁** 压缩"考量/权衡"段 —— 五段式剧情点中，② 考量/权衡 是灵魂段，不可省略
9. **严禁** AI 在同一角色下引用上一个角色的推理过程（见信标切换协议）
10. **★ 🎯 主智能体严禁** 读取子智能体产出文件的内容——只检查文件存在性和 PASS/FAIL
11. **★ 🎯 主智能体严禁** 直接编辑任何创作文件 —— 只能写 main-log.md
12. **★ 严禁** 子智能体之间直接通信 —— 所有通信通过文件系统 + 🎯 主智能体中转
13. **★ 严禁** 子智能体读取其他子智能体的推理过程 —— 只读取最终产出文件
14. **★ 严禁** 子智能体编辑其他子智能体的产出文件 —— 只写自己的输出目录
15. **★ 严禁** 在没有 🎯 主智能体分配的情况下自行开始工作

## 附录：关键术语表

| 术语 | 定义 |
|------|------|
| SKILL.md | 技能包顶层总控文件，定义角色索引、流程路由、全局规则 |
| skill | 一个工作流定义文件（outline.md / write.md / review.md 等） |
| 🎯 主智能体 | 编排者，负责分配任务、检查 PASS/FAIL、写 main-log.md |

| 框架层角色 | 人格固定，不随项目变化 |
| 元层角色 | 需用户激活后实例化，产出实例化文件 |
| NOVEL_ORCHESTRATION.md | 🆕 总文档，所有角色只读，定义全局规则和流程路由 |
| main-log.md | 🆕 🎯 主智能体写入的项目级日志 |
| 信标切换协议 | 角色切换时强制输出的冷启动声明，隔离上一个角色的推理记忆 |
| 上下文档案 | 用户设计的"静态设定+动态续写指令"记忆管理方案 |
| 0 号版本 | 🟢 项目启动官在初始化时生成的初始上下文档案 |
| 三线并行 | 每章必须同时推进主线、副线、人物线 |
| 五段式剧情点 | 危机/目标 → 考量/权衡（灵魂）→ 行动/决策 → 结果/转折 → 收益/情报 |
| 红队审计报告 | 🔴 红队检察官 输出的格式化审计结果 |
| 用户代理分级响应 | 🟢建议 / 🟡警示 / 🔴阻断 三级机制 |
