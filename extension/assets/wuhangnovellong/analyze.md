## 📊 步骤 1：故事梗概提取

| 项目 | 内容 |
| :--- | :--- |
| **输入** | 样板书 `.txt` 路径（步骤 0 已确认） |
| **输出** | `1-边界/1.1_全书故事梗概.md`（全书或开篇梗概） |

**⚠️ 执行前须先与用户确认**：本步有两种方式，**二选一**。在跑脚本或读原文之前，必须先明确询问用户：
- **方式 A**：用脚本批量生成全书梗概（需配置 API，有 Token 成本）；
- **方式 B**：由 Agent 读取样板书前 N 章生成开篇梗概（无需 API）。
待用户选择后，再按对应方式执行。无 API 或用户不想用脚本时用方式 B；后续步骤 3、4 可照常进行（基于开篇梗概 + 分段读原文）。

### 方式 A：脚本批量生成全书梗概（需 API）

按以下顺序在**对话中**完成，不要跳过步骤。

**1. 提醒用户注意**
- 样板书很长时 Token 消耗大（武行之前拆过自己的 200 章小说，花费 3–6 美金），可只处理部分章节或选用低成本 API。
- 使用 `.env` 配置 API 最安全：密钥仅存于本地文件、不会出现在命令行或聊天记录中，也不会被提交到仓库。

**2. 若用户尚未配置 API：先让脚本创建 `.env` 模板**
- 在**项目根目录**执行下面的命令（可由 AI 通过 Shell 代为执行一次）：
  ```bash
  python3 .soloent/skills/wuhangnovellong/scripts/summarize.py 1-边界/[书名].txt
  ```
- 脚本检测到缺少 API Key 时，会**自动在项目根创建 `.env` 文件**（模板）。
- 执行后，**在对话中明确提示用户**：「已在项目根创建 `.env` 文件，请打开该文件（路径：`项目根/.env`）查看并准备填写。」

**3. 提示用户打开并填写 `.env`**
- 在对话中说明：请打开项目根目录下的 **`.env`** 文件，在 `NOVEL_API_KEY=` 后面填入你的 API 密钥；如需更换服务商，可改 `NOVEL_API_BASE` 和 `NOVEL_API_MODEL`。**仅限本地使用，请勿将 `.env` 提交到仓库。**
- 请用户填写并保存后，在对话中回复「已填写」或「没问题」。

**4. 确认后执行脚本**
- 确认样板书已放入 `1-边界/` 下，且用户已确认 API 填写无误。
- **由 AI 在项目根代为执行脚本命令**（或给出命令供用户执行）：
  ```bash
  python3 .soloent/skills/wuhangnovellong/scripts/summarize.py 1-边界/[书名].txt
  ```
- 脚本会从项目根或 `.novelkit/.api.env` 读取 `.env` 配置，请勿在命令行中填写或粘贴 API Key。

**结果**：生成 `1-边界/1.1_全书故事梗概.md`。

### 方式 B：Agent 生成开篇梗概（无需 API）
- **执行**：不跑脚本，由 Agent 读取样板书前 N 章（如前 15 章或前 3000 行），在对话中生成开篇梗概。
- **输出**：写入 `1-边界/1.1_全书故事梗概.md`，可在文件中注明「开篇梗概，非全书」。后续若用户提供 API，可再跑脚本覆盖或追加为全书梗概。

### 步骤 1 完成后
- **须提示用户详细阅读 `1-边界/1.1_全书故事梗概.md`**，校验是否有明显断点或缺章，确认无误后再进入步骤 2。

## 📊 步骤 4：全书框架与节奏拆解

| 项目 | 内容 |
| :--- | :--- |
| **输入** | `1-边界/1.1_全书故事梗概.md`；可选：样板书章节标题列表（用于统计节奏） |
| **输出** | `1-边界/1.4_全书框架.md` |

- **输出格式（固定四大板块）**：

  1. **全书剧情单元总览**
     - 按【地图 + 身份/阶级 + 核心目标】复合维度划分阶段
     - 以表格形式呈现：章节区间、地图、身份、目标、故事梗概
     - **换地图逻辑**：分析为什么要换地图（战力崩坏/资源枯竭/主动扩张）

  2. **核心节奏公式**
     - 提炼出本书的标准循环公式
     - 例：发现危机→试错→重开→布局→收网
     - 总结可复用的叙事节奏模板

  3. **换地图逻辑分析**
     - 作者是何时换地图的？（主动 vs 被动）
     - 换地图的前兆是什么？（资源枯竭/战力崩坏/追杀）

  4. **对新书的启示**
     - **开局节奏**：如何快速建立危机感
     - **中期布局**：如何进行多线操作
     - **后期格局**：如何实现身份跃迁与权限升级

- **禁止事项**：不要画曲线图等冗余内容，专注以上四大核心板块。

## 🔍 步骤 5：微观节奏与剧情单元拆解

| 项目 | 内容 |
| :--- | :--- |
| **输入** | 样板书指定章节正文（前三章 + 一个完整高潮单元约 3–10 章）、可选 `1.1_全书故事梗概.md` 用于选段 |
| **输出** | `1-边界/1.5_微观节奏拆解.md` |

### 子步骤 5.1：单剧情单元拆解

#### 目标
分析作者如何用几章的篇幅讲好一个完整的小故事（小高潮）。

#### 选材方法
1. **先看梗概**: 浏览 `1.1_全书故事梗概.md`
2. **寻找样本**: 字数占比适中（约3-6章）、起承转合明显的完整事件
3. **推荐样本**: 优先选择“新手村第一次大危机/大收获”（通常在第10-20章左右）
4. **提取正文**: 根据梗概锁定的章节号，去样板书原文中提取对应章节

#### 分析模型：标准网文四步法

**1. 起 (Setup)**
- 欲望与阻碍
- 主角想要什么？谁在阻拦？
- （拉仇恨阶段）

**2. 承 (Build-up)**
- 试探与压抑
- 主角尝试解决但失败，或者被进一步羞辱/打压
- （情绪积蓄阶段）

**3. 转 (Climax)**
- 爆发与反转
- 金手指介入，或者智慧破局，形成最大反差
- （释放阶段）

**4. 合 (Resolution)**
- 收获与余波
- 战利品清点，地位提升，引出下一个麻烦
- （落袋为安阶段）

---

### 子步骤 5.2：单章精密拉片——以「黄金三章」为例

#### 目标
显微镜级分析。针对前三章，看作者是如何留住读者的。

#### 输入
- 样板书第 1、2、3 章（逐章分析）

#### 分析维度

**1. 切入点**
- 第一段写了什么？
- 环境/对话/危机/金手指？

**2. 信息密度控制**
- 什么时候抛出世界观？
- 什么时候抛出设定？

**3. 场景结构分析**
- **场景组合**: 单章通常包含几个场景？
  - 例：1个大对话+1个小动作+1个内心独白
- **场景写法**: 每个场景侧重什么？
  - 心理博弈/环境渲染/动作描写/信息交代
- **镜头语言**: 作者如何切换场景？
  - 空镜头转场/时间跳跃/“镜头”拉远拉近
- **场景缝合**: 上一段还在家里，下一段怎么去学校的？
  - 硬切 vs 软切
- **视角游移**: 为了展示主角的强/惨，视角是否切换到了旁人身上？

**4. 断章钩子**
- 这一章结尾最后一句话是什么作用？

### 步骤 5 完成后
- **须提示用户详细阅读 `1-边界/1.5_微观节奏拆解.md`**，确认无误后再进入步骤 6。

## 🧩 步骤 6：用户自定义拆解

> **须提示用户**：在完成步骤 1–5 之后，主动询问用户是否需要进行**自定义拆解**。若用户需要，可说明要拆解的章节范围、维度（如感情线、反派线、打斗、伏笔等）及输出路径，由 Agent 按约定执行并输出到指定位置。

### 输入（与用户交互确定）
- **拆解范围**：例如「第 50–80 章」「仅前三章」「全书所有打斗场面」。
- **拆解维度**：用户自选或组合，例如：
  - 仅感情线（谁和谁的关系变化、关键事件）
  - 仅反派/配角成长线
  - 仅打斗/战斗场面写法
  - 仅金手指使用与升级节点
  - 仅伏笔埋设与收回
  - 或用户用自然语言描述（如「我想拆这本书的节奏，看每章结尾怎么留钩子」）。
- **输出位置**：用户指定，例如 `1-边界/自定义_感情线拆解.md`、`1-边界/自定义_打斗场面.md`，或默认 `1-边界/自定义_[描述].md`。

### 动作流程
1. **询问**：在标准拆解结束或用户提出「我还想拆 XXX」时，主动问：
   - 「需要做**自定义拆解**吗？请说明：① 要分析的章节或内容范围；② 想拆的维度（如感情线、反派线、打斗、伏笔等）；③ 输出文件名或路径（可省略则用默认）。」
2. **确认**：复述用户的范围、维度、输出路径，确认无误后再执行。
3. **执行**：Agent 按约定读取样板书相应部分，按用户要求的维度进行分析，生成结构化内容（建议用标题、列表或表格），写入用户指定路径。

### 步骤 6 生成文件后
4. **须提示用户详细阅读该文件**，确认无误后结束或继续其他步骤。自定义拆解文件为参考文件，**无需登记到 SOLOENT**；若用户希望被总控引用，可自行在 SOLOENT 中添加指针（非本流程必须）。

### 输出
- 用户指定的文件，内容为约定维度的拆解结果；若用户未指定路径，则使用 `1-边界/自定义_[简短描述].md`。

### 示例
- 用户：「帮我拆 50–80 章的感情线，输出到 `1-边界/自定义_感情线.md`。」  
  → Agent 读取 50–80 章，提取感情相关情节与转折，写入 `1-边界/自定义_感情线.md`。
- 用户：「我只关心反派怎么一步步变强的，拆一下。」  
  → Agent 确认范围（全书或某几卷）、输出路径，然后按「反派成长线」维度拆解并写入指定文件。
